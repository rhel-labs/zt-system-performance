= Resolution

== Fixing the Problem

Now that we've thoroughly diagnosed the issue, it's time to resolve it. We need to both stop the service immediately AND prevent it from starting again at boot.

[IMPORTANT]
====
When resolving service-related issues, remember:

* `systemctl stop` - Stops the service immediately (but it may restart at boot)
* `systemctl disable` - Prevents automatic startup at boot (but doesn't stop it now)
* You typically need BOTH to fully resolve the issue
====

== Step 1: Stop the Service

First, let's stop the service to immediately free up system resources:

[source,bash,role=execute]
----
sudo systemctl stop system-optimizer.service
----

The service should stop immediately. Let's verify:

[source,bash,role=execute]
----
systemctl status system-optimizer.service
----

You should see that the service is now "inactive (dead)". The processes are gone and resources are freed.

== Step 2: Disable the Service

Stopping the service is only half the solution. Remember that the service was *enabled*, meaning it would start automatically at the next boot. We need to disable it:

[source,bash,role=execute]
----
sudo systemctl disable system-optimizer.service
----

This removes the service from automatic startup. Let's verify:

[source,bash,role=execute]
----
systemctl is-enabled system-optimizer.service
----

This should now return "disabled" or an error indicating the service won't start at boot.

[TIP]
====
If you wanted to completely remove the service, you could also delete the unit file at `/etc/systemd/system/system-optimizer.service` and run `sudo systemctl daemon-reload`. However, keeping it disabled is often safer - it preserves evidence of what happened and allows investigation later.
====

== Step 3: Verify System Recovery

Now let's confirm that the system has recovered. Check the load average again:

[source,bash,role=execute]
----
uptime
----

The load should now be much lower, close to or below the number of CPU cores.

Check memory:

[source,bash,role=execute]
----
free -h
----

Available memory should have increased significantly.

Finally, check what's consuming resources now:

[source,bash,role=execute]
----
top -b -n 1 | head -20
----

The stress-ng processes should be completely gone, and CPU idle time (%id) should be much higher.

== Step 4: Document What Happened

Create a brief incident report for the team:

[source,bash,role=execute]
----
cat > /home/rhel/incident-report.txt << 'EOF'
INCIDENT REPORT: System Performance Issue - RESOLVED

Date: $(date)
Reported by: Users experiencing slow system performance

Problem:
- High CPU and memory usage
- System load significantly above normal
- Service system-optimizer.service consuming resources

Root Cause:
- Manager Scott installed "system-optimizer.service"
- Service was running stress-ng (a stress testing tool)
- Scott misunderstood the tool's purpose - it CONSUMES resources to test
  system capacity, rather than optimizing performance

Resolution:
- Stopped the service: systemctl stop system-optimizer.service
- Disabled automatic startup: systemctl disable system-optimizer.service
- System resources returned to normal levels

Lessons Learned:
- Always test new services in non-production environments first
- Verify what a service actually does before deploying it
- Just because something has "optimizer" in the name doesn't mean it optimizes

Status: RESOLVED
EOF
----

View the report:

[source,bash,role=execute]
----
cat /home/rhel/incident-report.txt
----
